# 65行で動く「ブログ記事執筆ツール」のコードを、ゼロから読んでみた

こんにちは。今日は社内の AI ライティングツールの中から、`tools/blog.py` という1ファイルを取り上げます。たった65行ですが、「画面の作り方」「AIへの指示の作り方」「アプリへの組み込み方」がきれいに詰まっていて、**このアプリの“ツール1個分”のお手本**になっています。順番に読み解いていきましょう。

## まず全体像：このファイルは2つのことしかしていない

細かく見る前に地図を持っておくと迷いません。このファイルがやっているのは、実はたった2つです。

1. **`render()`** … 画面（入力欄やボタン）を描いて、ボタンが押されたらAIを呼ぶ
2. **`SPEC`** … 「このツールはこういう名前ですよ」とアプリ本体に自己紹介する

この「`render()` と `SPEC` を用意する」というのが、このアプリで**全ツール共通のお約束**になっています。だから新しいツールを足したいときも、この2つを書けばOK、というわけですね。

## 1. 冒頭の `import` と「AIの人格設定」

```python
import streamlit as st
from tools.common import ToolSpec, run_generation, show_result_actions
```

- `streamlit`（`st`）は、**Pythonだけで画面（Webアプリ）を作れる道具**です。`st.text_input(...)` と書くだけで入力欄が1つ出ます。HTMLもJavaScriptも書きません。
- `tools.common` からは、**他のツールと共通で使う部品**を3つ借りています。これが今回の主役級です。
  - `run_generation` … AIに文章を作ってもらう係
  - `show_result_actions` … 生成結果に「文字数」「再生成」「保存」を付ける係
  - `ToolSpec` … ツールの自己紹介カードのひな型

次に、こんな文章の定数があります。

```python
SYSTEM = (
    "あなたはプロのWebライター兼SEO編集者です。読者にとって読みやすく、"
    "検索エンジンにも評価される、自然な日本語のブログ記事を書きます。"
    "見出し(Markdownの##, ###)を適切に使い、結論ファーストで構成します。"
)
```

これは **AIに与える「役割（キャラ設定）」** です。専門用語で *システムプロンプト* と呼びます。
ポイントは、ユーザーが入力する内容（テーマなど）とは別に、**「あなたはプロのWebライターです」という土台を最初に固定しておく**こと。こう書いておくと、AIは毎回プロのライターになりきって答えてくれます。

## 2. `render()` の前半：入力フォームを並べる

ここからが画面づくりです。1行ずつ、ほぼ「書いた順に上から画面に出る」と思って読めます。

```python
st.subheader("✍️ ブログ記事執筆")
st.caption("テーマと条件を入力すると、構成済みのブログ記事を生成します。")

topic = st.text_input("記事のテーマ・タイトル", placeholder="例：在宅ワークで集中力を保つ方法")
keywords = st.text_input("含めたいキーワード（任意・カンマ区切り）", ...)
```

- `st.subheader` / `st.caption` … 見出しと小さな説明文。
- `st.text_input(...)` … **1行の入力欄**。ここがミソで、`topic = st.text_input(...)` のように受け取ると、**ユーザーが打ち込んだ文字がそのまま `topic` という変数に入ります**。`placeholder` は薄いグレーで出る入力例で、実際の値ではありません。

次に、横並びのレイアウトが出てきます。

```python
col1, col2, col3 = st.columns(3)
with col1:
    tone = st.selectbox("トーン", ["丁寧・親しみやすい", "プロフェッショナル", ...])
with col2:
    length = st.selectbox("文字数の目安", ["約800字", "約1500字", "約3000字"])
with col3:
    audience = st.text_input("想定読者（任意）", ...)
```

`st.columns(3)` は **画面を縦3列に分ける**おまじない。`with col1:` のブロックに書いたものが1列目に……という形で、3つの設定を横に並べています。
`st.selectbox` は**プルダウンの選択肢**で、選ばれた値が `tone` や `length` に入ります。

最後にチェックボックス。

```python
outline_only = st.checkbox("まず構成案（見出しだけ）を作る", value=False)
```

チェックが入れば `outline_only` が `True`、入っていなければ `False`。後でこの真偽でAIへの指示を切り替えます。

## 3. `render()` の中盤：ボタンが押されたらAIに渡す「指示書」を組み立てる

ここがこのファイルの心臓部です。

```python
if st.button("記事を生成", type="primary", disabled=not topic):
```

`st.button(...)` は **押された瞬間だけ `True` を返す**ので、`if` の中身は「ボタンが押されたとき」だけ動きます。
注目は `disabled=not topic`。**テーマ（`topic`）が空っぽのうちはボタンを押せない**ようにしています（`not 空文字` は `True` → 無効）。地味ですが「何も入れずに生成」を防ぐ親切設計です。

ボタンが押されたら、まずチェックボックスで指示を分岐します。

```python
if outline_only:
    instruction = "記事の見出し構成（アウトライン）だけを…本文は書かないでください。"
else:
    instruction = f"上記の条件で、{length}程度のブログ記事を本文まで完成させてください。"
```

「見出しだけ」なら骨組みだけ、そうでなければ `{length}`（例：約1500字）を埋め込んで本文まで書かせる、という違いです。

そして本番、**AIへの指示書（プロンプト）を文章として組み立てます**。

```python
prompt = f"""次の条件でブログ記事を作成してください。

# テーマ
{topic}

# 含めたいキーワード
{keywords or "指定なし"}
...
"""
```

これは `f"""..."""` という **f文字列**で、`{topic}` のような波かっこの中に**変数の中身が差し込まれます**。さっきユーザーが入力した値が、ここで全部1枚の指示書に集約されるわけです。

ちょっとした名脇役が `{keywords or "指定なし"}`。
これは **「キーワードが入力されていればそれを、空なら『指定なし』を使う」** というPythonらしいイディオムで、AIに空欄を渡さない工夫です。`audience` 側の `or "一般読者"` も同じ役割。

## 4. `render()` の後半：AIを呼んで、結果に機能を付ける

指示書ができたら、あとは共通部品に丸投げします。

```python
run_generation(prompt, system_instruction=SYSTEM, temperature=0.8, state_key="blog_result")
```

- `prompt` … さっき組み立てた指示書
- `system_instruction=SYSTEM` … 冒頭の「プロのWebライター」設定
- `temperature=0.8` … **AIの“奔放さ”ダイヤル**。0に近いほど堅実・無難、1に近いほど自由で創造的。ブログは多少のびのび書いてほしいので 0.8 とやや高め（ちなみに翻訳や要約のツールはもっと低い値で“忠実”寄りにしています）。
- `state_key="blog_result"` … **生成結果に貼る名札**。他のツールの結果と混ざらないよう、ツールごとに違う名前を付けています。

そして最後の1行。

```python
show_result_actions("blog_result")
```

これは**ボタンの `if` の外**にあるのがポイント。生成した直後だけでなく、画面を再表示するたびに「`blog_result` という名札の結果があるなら、その下に〈文字数〉〈🔄再生成〉〈💾保存〉を出してね」と毎回確認しているのです。

> 💡 補足：`run_generation` と `show_result_actions` の中身は `tools/common.py` にあります。`blog.py` は「何を聞くか・何を指示するか」だけに集中し、「AIの呼び出し方・結果の見せ方」は共通部品に任せている——この**役割分担**が、コードを短く・読みやすく保っている理由です。

## 5. 最後の `SPEC`：アプリ本体への自己紹介

```python
SPEC = ToolSpec(
    key="blog",                       # 内部用のユニークな識別子
    label="✍️ ブログ記事執筆",        # サイドバーに出る名前
    description="テーマからSEOを意識したブログ記事を自動生成します。",
    render=render,                    # ← さっき定義した画面描画の関数そのもの
)
```

`ToolSpec` は**ツールの“名刺”**です。注目は最後の `render=render`。
これは **関数そのものを渡している**（`render()` と呼び出しているのではない）のがコツで、「この名刺を持っているツールは、必要になったらこの `render` を実行して画面を出してね」とアプリ本体に託しているわけです。

アプリ本体（`app.py`）はこの名刺の `label` をサイドバーに並べ、選ばれたツールの `render` を呼ぶだけ。だから **`blog.py` を1枚書いて名刺を登録すれば、サイドバーに自動で仲間入り**します。

## まとめ：このファイルから学べる「ツールの型」

最後に流れをもう一度。

| 区分 | コード | 役割 |
|---|---|---|
| 役割設定 | `SYSTEM` | AIを「プロのWebライター」にする土台 |
| 入力 | `st.text_input` / `selectbox` / `checkbox` | ユーザーの希望を変数に集める |
| 組み立て | `prompt = f"""..."""` | 入力を1枚の指示書にまとめる |
| 実行 | `run_generation(...)` | AIに渡して文章を生成（共通部品） |
| 後処理 | `show_result_actions(...)` | 文字数・再生成・保存を付ける（共通部品） |
| 登録 | `SPEC = ToolSpec(...)` | アプリ本体への自己紹介 |

**入力を集める → 指示書を作る → 共通部品に渡す → 名刺で登録する。** これがこのアプリのツール1個の“型”です。`blog.py` が読めれば、`email_reply.py` も `summarize.py` も、骨格は全部同じなのですぐ読めるはずです。次に別のツールを開いて、「あ、これも同じ型だ」と確認してみると、理解が一気に定着しますよ。
